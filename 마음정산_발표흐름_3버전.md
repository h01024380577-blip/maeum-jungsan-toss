# 마음정산 발표 흐름 — 3가지 버전 제안

> 평가기준: 문제 정의(25%) · 문제 해결의 타당성(25%) · 창의성(25%) · 발표(25%)
> 공통 원칙: ① 파싱은 "멀티모달 AI"로 정확히 표현(OCR ❌) ② 금액 추천은 "규칙 기반 + 데이터 기반 고도화 로드맵"으로 솔직하게 ③ 제품명값을 회수하는 **정산 결과 데모**를 반드시 포함 ④ 창의성을 독립된 장면으로 노출 ⑤ 감정적 한 문장으로 클로징

각 슬라이드 표기: **[시간]** · 핵심 메시지 · 말할 내용 · *(찍는 평가기준)*

---

## 버전 A — 실행·그로스 중심 (Execution-Driven)

**컨셉:** "우리는 만들었고, 출시했고, 사용자를 모았고, 피드백으로 개선했다." 실제 트랙션과 운영 경험을 무기로 삼는 버전.
**적합 상황:** 심사위원이 실행력·완성도·사업성을 중시할 때. 데모가 안정적으로 동작할 때.
**총 길이:** 약 6분

| # | 슬라이드 | 내용 |
|---|---|---|
| 1 | **[0:20] 문제 정의 영상** | 경조사비 갈등 영상. "얼마 해야 하지? 작년에 내가 얼마 했더라?" 누구나 겪은 장면으로 후킹. *(문제정의)* |
| 2 | **[0:30] 시장분석 — 문제 검증** | 기존 경조사 관리 앱들의 구성(단순 기록·가계부형) + 경조사 문화 변화(비대면 청첩장·계좌 송금 보편화). → **두 병목 도출: 직접 입력 / 금액 결정.** *(문제정의)* |
| 3 | **[0:15] 서비스 한 줄 소개** | "초대장 링크만 붙이면 자동 입력되고, 관계별로 주고받은 마음이 정산되는 앱 — 마음정산." *(해결 타당성 도입)* |
| 4 | **[0:50] 차별화 기능 + 라이브 데모 ①·②** | 실제 UI로: (a) 청첩장 URL 붙여넣기 → 3초 자동 입력, (b) **관계별 받은/보낸 마음 정산 화면**. 제품명값 회수. *(해결 타당성 + 발표)* |
| 5 | **[0:30] 왜 AI인가 (타당성 검증)** | "포맷이 제각각인 비정형 초대장은 규칙으로 못 푼다 → 멀티모달 AI로 의미까지 읽어 자동 입력. 금액 추천은 현재 관계·이력 기반 규칙, 데이터가 쌓이면 학습으로 고도화." *(해결 타당성)* |
| 6 | **[0:40] 아키텍처 + 개발 전략 (압축 1장)** | 다이어그램 1장. EC2 단일 배포로 **빠른 출시 우선**(피드백 빠르게 회수) + 앱인토스 선택 이유(토스 유저베이스로 마케팅 부담↓, 송금 연계, 초창기 락인). *(해결 타당성 + 창의성)* |
| 7 | **[0:50] 실행 스토리: 어떻게 실사용자를 만들었나** | "출시했지만 사용자 0 → 피드백을 못 받음"이 첫 위기. Threads·텔레그램·지인 홍보로 적극 마케팅 → 대시보드 성장 그래프 → 첫 피드백 발생. **능동형 제목으로 그릿을 강조.** *(발표 + 해결 타당성)* |
| 8 | **[0:40] v1 → v2 개선 루프** | "v1은 이랬고, 이런 피드백이 왔고, v2로 이렇게 고쳤다" 구조. 예: 토스 검색 노출 위해 앱 이름을 '경조사 가계부'로 변경(SEO). 출시 초기라 잦은 업데이트에도 이탈 적다는 판단. *(해결 타당성 + 발표)* |
| 9 | **[0:30] 비즈니스 모델** | AI 분석 한도를 크레딧으로 관리 → 광고 시청으로 충전 → 서버·AI 호출 비용을 광고로 충당. 지속 가능 구조. *(해결 타당성)* |
| 10 | **[0:30] 향후 발전 + 창의 비전** | 경조사비 비용처리(세금 절감 수요) → 초대장 이미지 S3 보관, QR로 축의/부의금 수금까지 → **경조사비 전체 흐름을 앱 하나로** → 받은 데이터까지 쌓여 추천 정교화 선순환. *(창의성)* |
| 11 | **[0:15] 클로징 — 감정적 한 문장** | "경조사는 결국 관계의 기록입니다. 마음정산은 그 마음을 잊지 않게 합니다." *(발표)* |

**강점:** 실행·운영 서사가 가장 두껍다. 트랙션으로 신뢰를 산다.
**주의:** 슬라이드가 많아 시간 초과 위험 → 6·9는 각 1장·40초 이내로 압축. 창의성(③)이 뒤로 몰리니, 4번 데모에서 "전체 루프"를 한 줄 예고해 미리 심어둘 것.

## 버전 B — 문제·창의성 중심 (Insight-Driven)

**컨셉:** "이 문제는 생각보다 깊고, 우리의 관점은 기존과 다르다." 문제의 본질과 독창적 비전으로 승부하는 버전.
**적합 상황:** 심사위원이 창의성·인사이트·확장 가능성을 높게 볼 때. 아이디어/기획 비중이 큰 대회.
**총 길이:** 약 5분 30초

| # | 슬라이드 | 내용 |
|---|---|---|
| 1 | **[0:25] 문제 정의 영상** | 경조사비 갈등 영상. 감정적 몰입으로 시작. *(문제정의)* |
| 2 | **[0:40] 문제의 재정의 — 본질 파고들기** | 표면 문제는 "기록 귀찮음"이지만 본질은 **"관계의 주고받음을 추적·정산하지 못함"**. 기존 가계부 앱은 *기록*만 하고 *관계 단위 정산*을 못 한다 → 진짜 페인은 여기. *(문제정의 + 창의성)* |
| 3 | **[0:40] 시장분석 — 문화 변화가 만든 틈** | 비대면 청첩장·계좌 송금 보편화로 데이터는 디지털인데, 그걸 정리하는 도구는 아날로그(메모·캘린더)에 머묾. 입력 병목 + 금액 결정 병목을 데이터로 제시. *(문제정의)* |
| 4 | **[0:20] 서비스 한 줄 소개** | "관계의 마음을 자동으로 기록하고 정산하는 앱, 마음정산." *(전환)* |
| 5 | **[0:45] 창의 포인트 — 우리만의 관점 (독립 1장)** | 세 가지 차별 발상: (a) **초대장 링크 파싱**이라는 새 진입점, (b) 부고장 자동 테마 전환 등 맥락 감응 UX, (c) **경조사비 전체 루프**(보내기 → 정산 → QR 수금)를 한 앱으로 닫겠다는 비전. *(창의성)* |
| 6 | **[0:50] 데모로 증명 ①·②** | 실제 UI: 자동 입력 데모 + **정산 결과 화면** 데모. 관점이 말이 아니라 동작하는 제품임을 증명. *(해결 타당성 + 발표)* |
| 7 | **[0:30] 왜 AI인가 + 기술 타당성** | 비정형 초대장 → 멀티모달 AI 파싱 필연성. 추천은 규칙 기반 → 데이터 학습 로드맵. 아키텍처 다이어그램 한 장 곁들임. *(해결 타당성)* |
| 8 | **[0:30] 어떻게 빠르게 검증했나** | EC2 몰빵으로 빠른 출시 → 앱인토스로 유저베이스·송금연계 확보 → 마케팅으로 실사용자·피드백 → v1/v2 개선(앱 이름 SEO 변경 등). 실행은 "관점을 검증하는 수단"으로 압축. *(해결 타당성)* |
| 9 | **[0:25] 비즈니스 모델 + 확장 선순환** | 광고-크레딧으로 비용 충당. QR 수금이 더해지면 받은 데이터까지 쌓여 추천 정교화 → 데이터 해자. *(창의성 + 해결 타당성)* |
| 10 | **[0:15] 클로징 — 감정적 한 문장** | "경조사는 관계의 기록입니다. 우리는 그 관계를 정산합니다." *(발표)* |

**강점:** 창의성(③)이 독립 비트(5번)로 가장 선명하다. 문제 정의가 깊다.
**주의:** 실행 트랙션(8번)이 압축되니, 그 한 장에 "실사용자 수·업데이트 횟수" 같은 숫자 1~2개를 꼭 넣어 공허해 보이지 않게 할 것.

## 버전 C — 데모·임팩트 중심 (Show, Don't Tell)

**컨셉:** "설명보다 보여준다." 데모를 앞에 배치하고 군더더기를 덜어낸 짧고 강한 피치.
**적합 상황:** 발표 시간이 짧을 때(3~4분), 또는 임팩트·전달력 점수가 결정적일 때. 데모 안정성이 보장될 때.
**총 길이:** 약 4분

| # | 슬라이드 | 내용 |
|---|---|---|
| 1 | **[0:20] 공감 후킹 (영상 짧게)** | 경조사비 갈등 영상을 15초로 짧게. 곧바로 "이거, 앱으로 3초면 끝납니다." *(문제정의)* |
| 2 | **[0:50] 라이브 데모 먼저** | 말보다 시연: 청첩장 URL 붙여넣기 → 자동 입력 → **정산 화면**까지 한 흐름으로. 가장 강한 무기를 맨 앞에. *(발표 + 해결 타당성)* |
| 3 | **[0:30] 방금 본 게 왜 어려운 문제인가** | 데모로 놀란 직후 역으로 문제 설명: 포맷 제각각 초대장 입력 병목 + 금액 결정 병목 + 기존 앱은 정산 불가. 시장/문화 변화 한 줄. *(문제정의)* |
| 4 | **[0:30] 왜 AI인가 (핵심 타당성)** | 비정형이라 규칙으론 못 푼다 → 멀티모달 AI. 금액 추천은 규칙 기반 → 데이터로 고도화. 한 문장으로 압축. *(해결 타당성)* |
| 5 | **[0:25] 어떻게 만들고 키웠나 (1장)** | EC2 빠른 출시 + 앱인토스(유저베이스·송금·락인) + 마케팅으로 실사용자·피드백 → v1/v2 개선. 숫자 1~2개로 트랙션 증명. *(해결 타당성)* |
| 6 | **[0:25] 창의 비전 — 전체 루프** | QR 수금까지 더해 경조사비 보내기→정산→받기를 한 앱으로 닫고, 받은 데이터로 추천 정교화. 비용처리(세금) 확장 한 줄. *(창의성)* |
| 7 | **[0:15] 비즈니스 모델 (한 줄)** | 광고-크레딧으로 AI·서버 비용 충당, 지속 가능. *(해결 타당성)* |
| 8 | **[0:15] 클로징 — 감정적 한 문장** | "경조사는 관계의 기록입니다. 마음정산이 그 마음을 이어드립니다." *(발표)* |

**강점:** 전달력·임팩트 최강. 짧고 기억에 남는다. 데모로 시작해 몰입을 선점.
**주의:** 데모 실패 시 전체가 흔들림 → **사전 녹화 GIF 백업 필수.** 문제 정의(①)가 데모 뒤로 가니 3번에서 "왜 어려운가"를 충분히 짚어 ① 점수를 확보할 것.

## 세 버전 비교 요약

| 항목 | A 실행·그로스 | B 문제·창의성 | C 데모·임팩트 |
|---|---|---|---|
| 길이 | ~6분 | ~5분 30초 | ~4분 |
| 무게중심 | 트랙션·운영 | 인사이트·비전 | 시연·전달력 |
| 가장 센 평가기준 | 해결 타당성 | 창의성 | 발표 |
| 가장 보강 필요 | 창의성(뒤로 몰림) | 실행 트랙션(압축) | 문제정의(뒤로 밀림) |
| 추천 상황 | 사업성 중시 심사 | 기획·아이디어 대회 | 짧은 피치·임팩트 결정전 |

**공통 체크리스트:** ① 파싱=멀티모달 AI(OCR 아님) ② 추천=규칙+로드맵 ③ 정산 데모 포함 ④ 창의성 독립 노출 ⑤ 감정적 클로징 ⑥ 데모 GIF 백업.
